XML Basics for Odoo – Views, Data Files, and QWeb

XML is the language Odoo uses for almost everything that is not Python: view definitions, data records, QWeb reports, email templates, and portal pages are all written in XML files. Before you write your first module you need a solid grasp of XML fundamentals — syntax rules, the document tree, attributes versus element content, special character escaping, namespaces, and XPath. This page covers exactly that, keeping view content (form, list, kanban, search) for the Basic Views tutorial.

⏱️ 35 min 🎯 Beginner 📅 Updated 2026

What is XML?

XML (eXtensible Markup Language) is a format for describing and storing structured data as plain text. It was designed to be both human-readable and machine-readable. Unlike HTML, which tells a browser how to display data, XML describes what the data means.

Key points:

  • You define your own tags — there are no predefined tags like HTML's <p> or <div>
  • XML is self-describing: a <book> element containing <title> and <author> children makes its meaning clear without documentation
  • XML is the foundation of many technologies: SVG, RSS, SOAP, Office documents (.docx, .xlsx), and of course Odoo's entire configuration system

Odoo uses XML for four purposes:

PurposeExample files
View definitionsviews/sale_order_views.xml
Data records (menus, actions, security)views/menu.xml, security/ir.model.access.csv
QWeb templates (reports, emails, portal, website)report/sale_report.xml
Module manifest dependencies (indirect)__manifest__.py data list
Python 
# Python — structure is implicit in the code
book = {"title": "Clean Code", "author": "Robert C. Martin", "price": 39.99}
XML 
<!-- XML — structure and meaning are explicit in the markup -->
<book>
    <title>Clean Code</title>
    <author>Robert C. Martin</author>
    <price>39.99</price>
</book>
ℹ️
XML errors block module loading

Odoo will fail to load your module with a traceback if any XML file contains a syntax error. The error message names the file and line number. Get comfortable reading and writing valid XML before your first module.

XML Tree Structure

Every XML document is a tree. There is one root element, and every other element is a child, grandchild, or deeper descendant. This tree structure is called the document tree (or DOM tree).

XML 
<odoo>                          <!-- root -->
  <data>                        <!-- child of root -->
    <record id="r1">            <!-- child of data -->
      <field name="name">Library</field>
      <field name="active">True</field>
    </record>
    <record id="r2">
      <field name="name">Books</field>
    </record>
  </data>
</odoo>

Vocabulary:

  • Root element — the single top-level element; in Odoo XML files this is always <odoo>
  • Parent / child — direct containment relationship
  • Ancestor / descendant — any level up or down the tree
  • Sibling — elements that share the same parent
  • Leaf node — element with no children (just text content)

A real Odoo XML file with the tree labelled:

XML 
<?xml version="1.0" encoding="UTF-8"?>
<odoo>                                  <!-- root -->
    <data>
        <record id="view_book_list" model="ir.ui.view">
            <field name="name">library.book.list</field>    <!-- leaf -->
            <field name="model">library.book</field>        <!-- leaf -->
            <field name="arch" type="xml">
                <list>
                    <field name="name"/>                    <!-- self-closing leaf -->
                    <field name="author_id"/>
                </list>
            </field>
        </record>
    </data>
</odoo>

Why tree structure matters in Odoo: XPath expressions navigate this tree to find and modify specific nodes during view inheritance.

XML Syntax Rules

XML has strict, non-negotiable syntax rules. Unlike HTML browsers that silently fix broken markup, Odoo's XML parser raises a parse error and refuses to load the module if any rule is violated.

Rule 1 — Every tag must close

XML 
<!-- Valid: explicit close tag -->
<field name="name">Library</field>

<!-- Valid: self-closing (for empty elements) -->
<field name="name"/>

<!-- INVALID: missing close tag — module will fail to load -->
<field name="name">Library

Rule 2 — Tags are case-sensitive

XML 
<Record id="x">    <!-- This is a "Record" element -->
<record id="x">    <!-- This is a "record" element — completely different -->

Odoo always uses lowercase tag names. <Record> and <record> would be treated as two distinct element types.

Rule 3 — Attributes must be quoted

XML 
<!-- Valid -->
<record id="view_book" model="ir.ui.view">

<!-- INVALID: unquoted attribute values -->
<record id=view_book model=ir.ui.view>

Rule 4 — Exactly one root element

XML 
<!-- Valid: one root -->
<odoo>
    <data>...</data>
</odoo>

<!-- INVALID: two root elements side by side -->
<odoo>...</odoo>
<odoo>...</odoo>

Rule 5 — Elements must be properly nested

XML 
<!-- Valid: proper nesting -->
<group>
    <field name="name"/>
</group>

<!-- INVALID: interleaved tags -->
<group>
    <field name="name">
</group>
    </field>

Rule 6 — The XML declaration is optional but recommended

XML 
<?xml version="1.0" encoding="UTF-8"?>

Odoo XML files include this declaration. The encoding="UTF-8" tells the parser that the file uses UTF-8 character encoding, which supports all Unicode characters (important for multi-language Odoo deployments).

⚠️
Well-formed vs Valid

An XML document is well-formed if it follows all syntax rules above. It is valid if it also conforms to a schema (DTD or XSD) that defines which elements and attributes are allowed. Odoo XML files must be well-formed; Odoo's own internal schema (RNG files) validates the structure.

XML Elements

An element is the fundamental building block of XML. An element consists of an opening tag, optional content, and a closing tag.

Element types:

1. Element with text content (most Odoo field values):

XML 
<field name="name">Library Management</field>

2. Element with child elements:

XML 
<record id="action_books" model="ir.actions.act_window">
    <field name="name">Books</field>
    <field name="res_model">library.book</field>
</record>

3. Empty element (self-closing):

XML 
<field name="name"/>           <!-- same as <field name="name"></field> -->
<separator/>
<chatter/>

4. Element with mixed content (text and child elements — rare in Odoo):

XML 
<p>Click <a href="/contact">here</a> to contact us.</p>

Element naming rules:

  • Can contain letters, numbers, hyphens, underscores, periods
  • Must start with a letter or underscore — NOT a number or hyphen
  • Cannot contain spaces
  • Cannot start with "xml" (reserved)

Valid Odoo element names: <record>, <field>, <menuitem>, <template>, <t>, <xpath>

Nesting and parent-child relationships in a real Odoo view:

XML 
<form>                      <!-- parent -->
    <sheet>                 <!-- child of form, parent of group -->
        <group>             <!-- child of sheet, parent of field elements -->
            <field name="name"/>        <!-- child of group -->
            <field name="partner_id"/>  <!-- sibling of name field -->
        </group>
    </sheet>
</form>

XML Attributes

An attribute provides additional information about an element. Attributes always appear in the opening tag as name="value" pairs.

XML 
<record id="view_book_form" model="ir.ui.view" priority="16">
<!--      attribute 1         attribute 2        attribute 3 -->

Attribute rules:

  • Name-value pair, value always in quotes (single or double)
  • An element can have multiple attributes (no limit)
  • Each attribute name must be unique within an element
  • Attribute order does not matter to the XML parser
  • Attributes cannot contain child elements

Elements vs attributes — when to use which:

Use attributes for...Use elements for...
Metadata about the record (id, type, model)The actual data value
Short identifiers and flagsMulti-line or structured content
Values that won't have sub-structureValues that might need child elements

Odoo follows this consistently:

XML 
<!-- id and model are ATTRIBUTES — they identify and classify the record -->
<record id="view_book_form" model="ir.ui.view">

    <!-- name is an ATTRIBUTE on field — it names which field this is -->
    <!-- The value (library.book.form) is the ELEMENT content -->
    <field name="name">library.book.form</field>

    <!-- type="xml" is an ATTRIBUTE — metadata about how to parse the content -->
    <field name="arch" type="xml">
        ...
    </field>
</record>

Important Odoo attributes to recognise:

ElementKey attributesMeaning
<record>id, modelXML ID and target model
<field>name, type, ref, evalField name; parse type; record reference; Python expression
<menuitem>id, name, parent, action, sequenceMenu identity and hierarchy
<filter>string, name, domain, contextDisplay label, programmatic name, domain applied
<xpath>expr, positionXPath expression, insert position
<button>name, string, type, class, iconMethod name, label, type (object/action), CSS class

eval attribute — evaluates a Python expression instead of treating the value as a string:

XML 
<!-- Without eval: stores the string "True" -->
<field name="active">True</field>

<!-- With eval: stores the Python boolean True -->
<field name="active" eval="True"/>

<!-- Useful for lists, tuples, and expressions -->
<field name="groups_id" eval="[(4, ref('base.group_user'))]"/>

ref attribute — references another record by its XML ID:

XML 
<!-- ref resolves the XML ID to the database integer ID at load time -->
<field name="action_id" ref="action_library_book"/>
<field name="parent_id" ref="base.group_user"/>

XML Namespaces

An XML namespace prevents element name conflicts when combining XML from different sources. A namespace is declared with xmlns and is identified by a URI.

XML 
<!-- xmlns:xsi declares the XML Schema Instance namespace -->
<odoo xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance">

When you will see namespaces in Odoo:

  1. RNG validation files — Odoo's view schema files use namespaces internally (you don't write these)
  2. CDATA with HTML — some older Odoo modules declare an HTML namespace in QWeb templates
  3. SOAP/Web services — if you integrate Odoo with SOAP APIs, response XML uses namespaces heavily
  4. Office documents — if you generate .docx/.xlsx from Odoo, the OpenXML format uses namespaces

For day-to-day Odoo module development, you will rarely write namespaces yourself:

XML 
<odoo>
    <!-- No namespace needed — Odoo's XML parser knows the expected structure -->
    <record id="..." model="...">
        <field name="...">value</field>
    </record>
</odoo>
💡
Namespaces in practice

Namespaces are important in theory but rarely written by hand in Odoo module development. You will encounter them when reading Odoo's internal RNG schema files or integrating with SOAP web services. For now, knowing what xmlns means is enough.

Special Characters and Entities

XML reserves five characters as part of its markup syntax. If these appear in your data (not as markup), you must escape them using XML entities:

CharacterEntityWhere you'll see it in Odoo
<&lt;Domain comparisons: ('amount', '&lt;', 500)
>&gt;Greater-than in invisible expressions
&&amp;URLs with query strings, Python and as &amp;&amp;
"&quot;A quote character inside a double-quoted attribute
'&apos;A quote character inside a single-quoted attribute

Real examples from Odoo code:

XML 
<!-- Domain with less-than: raw < would break XML parsing -->
<filter string="Small Orders"
        domain="[('amount_total', '&lt;', 100)]"/>

<!-- Invisible attribute with 'and' written as &amp;&amp; in older Odoo versions -->
<!-- (Odoo 17+ uses Python-style expressions directly) -->
<field name="delivery_date"
       invisible="state != 'confirmed' &amp;&amp; type != 'delivery'"/>

<!-- URL with query string — & must be &amp; -->
<a t-attf-href="/shop/product/#{slug(product)}&amp;pricelist=#{pricelist_id}">
    View product
</a>

CDATA sections — when a block contains many special characters (like embedded code), wrapping in CDATA avoids escaping every character:

XML 
<!-- Without CDATA: every < > & must be escaped — messy -->
<field name="arch" type="xml">
    &lt;form&gt;...&lt;/form&gt;
</field>

<!-- With CDATA: characters inside are treated as literal text -->
<field name="code"><![CDATA[
    if order.amount_total > 1000 and order.state == 'confirmed':
        discount = order.amount_total * 0.05
        order.write({'discount_amount': discount})
]]></field>

CDATA syntax: <![CDATA[ ... ]]>. The only string that cannot appear inside CDATA is ]]> (the closing marker).

How Odoo Parses XML Files

When you install or upgrade an Odoo module, the server processes every XML file listed in __manifest__.py. Understanding this process helps you debug load-time errors.

The parsing sequence:

Python 
# What happens internally (simplified):
import lxml.etree as etree

with open('views/book_views.xml', 'rb') as f:
    tree = etree.parse(f)   # Step 1 — parse XML into element tree
    root = tree.getroot()   # <odoo> root element

for record_el in root.iter('record'):  # Step 2 — walk <record> elements
    model = record_el.get('model')     # read model attribute
    xml_id = record_el.get('id')       # read id attribute
    # ... create or update database row

Common parse errors and what causes them:

Error messageLikely cause
XMLSyntaxError: Opening and ending tag mismatchMissing or wrong closing tag
XMLSyntaxError: EntityRef: expecting ';'& used without escaping it as &amp;
XMLSyntaxError: attributes construct errorUnquoted attribute value
XMLSyntaxError: Extra content at the endTwo root elements in the file
ValueError: External ID not foundref="module.id" refers to a record that isn't loaded yet

File load order matters. If views/menus.xml references an action defined in views/actions.xml, then actions.xml must be listed FIRST in the manifest's data list:

Python 
'data': [
    'security/ir.model.access.csv',   # 1st: security (models must exist)
    'views/book_views.xml',           # 2nd: views
    'views/actions.xml',              # 3rd: actions (reference views)
    'views/menus.xml',                # 4th: menus (reference actions)
],

XPath in Odoo View Inheritance

XPath (XML Path Language) is a query language for selecting nodes in an XML document. In Odoo, XPath is used exclusively for view inheritance — modifying an existing view from another module without touching the original module's code.

XPath expression anatomy:

XML 
<!--    //   field   [@name='partner_id']
         |    |       |
         |    |       predicate (filter by attribute value)
         |    element name to match
         double slash = search anywhere in the document
                        (single slash = search from root only)  -->

Axis shortcuts used in Odoo:

ExpressionMatches
//field[@name='x']Any <field> element with name attribute equal to x
//group[@name='g']Any <group> with name attribute g
//notebookThe first <notebook> element anywhere
//page[@name='p']A <page> with name p inside a notebook
//button[@string='Save']A <button> whose string attribute is Save
//sheet/group[1]The first <group> directly inside <sheet>
//div[@class='oe_title']A <div> with that CSS class

Position values — what happens at the matched element:

PositionEffect
afterInsert new XML immediately after the matched element
beforeInsert new XML immediately before the matched element
insideAppend new XML as the last child of the matched element
replaceRemove the matched element and replace with new XML
attributesModify attributes of the matched element (use <attribute> children)

Full working example — extending the sale order form:

XML 
<record id="view_sale_order_form_inherit_mymodule" model="ir.ui.view">
    <field name="name">sale.order.form.mymodule</field>
    <field name="model">sale.order</field>
    <!-- inherit_id points to the view being extended -->
    <field name="inherit_id" ref="sale.view_order_form"/>
    <field name="arch" type="xml">

        <!-- 1. Add a custom field after partner_id -->
        <xpath expr="//field[@name='partner_id']" position="after">
            <field name="x_project_code" string="Project Code"/>
        </xpath>

        <!-- 2. Add a new tab to the notebook -->
        <xpath expr="//notebook" position="inside">
            <page string="Project Details" name="project_details">
                <group>
                    <field name="x_project_manager_id"/>
                    <field name="x_deadline"/>
                </group>
            </page>
        </xpath>

        <!-- 3. Hide a field using attribute modification -->
        <xpath expr="//field[@name='client_order_ref']" position="attributes">
            <attribute name="invisible">1</attribute>
        </xpath>

        <!-- 4. Replace a field entirely -->
        <xpath expr="//field[@name='payment_term_id']" position="replace">
            <field name="x_custom_payment_term" string="Payment"/>
        </xpath>

    </field>
</record>

Shorthand syntax — for simple cases targeting a single field by name, Odoo also accepts:

XML 
<!-- Shorthand: use the element directly as the locator -->
<field name="partner_id" position="after">
    <field name="x_project_code"/>
</field>

Use XPath for non-field elements, duplicate field names, or any case where the shorthand might match the wrong element.

💡
Always name your groups and pages

Always add a name attribute to <group>, <page>, and <notebook> elements you create. Without a name, future modules can only target them with fragile positional XPath like //sheet/group[2] which breaks if any other module adds a group. With a name: //group[@name='my_group'] is stable.

XML Validation

There are two levels of XML correctness:

Well-formed — the document follows all XML syntax rules (tags close, attributes quoted, one root, etc.). A well-formed document can be parsed without error.

Valid — the document is well-formed AND conforms to a schema that defines which elements and attributes are permitted and where.

How Odoo validates XML files:

Odoo uses RNG (RELAX NG) schema files to validate view XML. These schemas are in odoo/addons/base/views/ and define what elements are allowed inside <form>, <list>, <kanban>, etc.

When you run Odoo with --dev=xml (development mode), view XML is validated against these schemas. During development, always test with --dev=xml to catch schema violations early:

Bash 
# Run Odoo with XML validation enabled
python odoo-bin -c odoo.conf --dev=xml

# Or upgrade a specific module with validation
python odoo-bin -c odoo.conf -u my_module --dev=xml

Validating XML files locally — before loading them into Odoo, check XML syntax using xmllint:

Bash 
# Check if a file is well-formed XML
xmllint --noout views/book_views.xml

# Output: nothing (success) or error message with line number
Bash 
sudo apt install libxml2-utils    # Ubuntu/Debian
brew install libxml2              # macOS

QWeb — Odoo's XML Template Engine

QWeb is Odoo's template engine built on top of XML. It adds dynamic behaviour to otherwise static XML through t-* directive attributes. QWeb is used for PDF reports, portal pages, website pages, and email templates.

DirectivePurposeExample
t-esc="expr"Output HTML-escaped value<t t-esc="order.name"/>
t-raw="expr"Output raw HTML (unsafe)<t t-raw="body_html"/>
t-if="condition"Render only if condition is true<div t-if="show_price">
t-elif="condition"Else-if branch<div t-elif="on_sale">
t-elseElse branch<div t-else="">
t-foreach="list" t-as="item"Loop<t t-foreach="lines" t-as="line">
t-call="template.xmlid"Include another template<t t-call="web.html_container">
t-set="var" t-value="expr"Set a variable<t t-set="total" t-value="sum(lines)">
t-att-href="expr"Dynamic attribute<a t-att-href="url">
t-attf-class="literal #{expr}"Dynamic attribute with string format<div t-attf-class="badge-#{state}">
t-field="record.field_name"Render field with Odoo formatting<span t-field="order.amount_total"/>

A minimal QWeb report template:

XML 
<template id="report_book_document">
    <t t-call="web.html_container">
        <t t-foreach="docs" t-as="doc">
            <div class="page">
                <h2><t t-esc="doc.name"/></h2>
                <table class="table">
                    <thead>
                        <tr><th>Author</th><th>Price</th></tr>
                    </thead>
                    <tbody>
                        <tr>
                            <td t-if="doc.author_id">
                                <t t-esc="doc.author_id.name"/>
                            </td>
                            <td t-else="">Unknown</td>
                            <td><t t-esc="doc.price"/></td>
                        </tr>
                    </tbody>
                </table>
            </div>
        </t>
    </t>
</template>

The <t> element is QWeb's invisible wrapper — it applies directives without adding an actual DOM element to the output.

Data and Demo Files

Every XML file Odoo loads must be declared in __manifest__.py. The manifest has two lists:

Python 
{
    'name': 'Library Management',
    'data': [
        'security/ir.model.access.csv',   # security rules (CSV, not XML)
        'data/res_config_settings.xml',   # default settings
        'views/book_views.xml',           # view definitions
        'views/menus.xml',                # menu items
        'report/book_report.xml',         # QWeb PDF report
        'data/mail_template.xml',         # email templates
    ],
    'demo': [
        'demo/demo_books.xml',            # sample data (learning/testing)
        'demo/demo_members.xml',
    ],
}

data files — loaded on every module install AND every upgrade. Use for view definitions, menus, actions, email templates, sequences, and any configuration record that defines how the module works.

demo files — loaded ONLY when the database was created with demo data enabled. Use for sample records and test fixtures.

noupdate="1" — wrap records in <data noupdate="1"> when they should be created once and never overwritten on upgrade:

XML 
<odoo>
    <!-- These records update on every upgrade (default) -->
    <record id="view_book_form" model="ir.ui.view">
        ...
    </record>

    <!-- These records are created once, never overwritten -->
    <!-- Use for: email templates, sequences, default values users may modify -->
    <data noupdate="1">
        <record id="sequence_library_book" model="ir.sequence">
            <field name="name">Library Book Sequence</field>
            <field name="code">library.book</field>
            <field name="prefix">LIB/%(year)s/</field>
            <field name="padding">4</field>
        </record>
    </data>
</odoo>

Load order is critical. Odoo processes files in the order listed in the manifest. The safe order is always: security → models/views → actions → menus → data.

External IDs (XML IDs)

Every <record> element must have a unique id attribute within the module. This is its external identifier (XML ID). Odoo stores these in the ir.model.data table and uses them to reference records reliably across modules — without depending on the database integer ID, which varies between installations.

Format:

  • Within a module: use just id → ref="action_library_book"
  • From another module: use module_name.id → ref="library_management.action_library_book"
XML 
<!-- Defining an external ID -->
<record id="action_library_book" model="ir.actions.act_window">
    <field name="name">Books</field>
    <field name="res_model">library.book</field>
</record>

<!-- Referencing within the same module -->
<menuitem id="menu_library_books"
          name="Books"
          action="action_library_book"/>

<!-- Referencing from another module (must prefix module name) -->
<menuitem id="menu_library_books_ext"
          name="Books"
          action="library_management.action_library_book"/>

Accessing external IDs in Python:

Python 
# Get the record object
action = self.env.ref('library_management.action_library_book')

# Get just the integer database ID
action_id = self.env.ref('library_management.action_library_book').id

# Check if an XML ID exists (won't raise if missing)
action = self.env.ref('library_management.action_library_book', raise_if_not_found=False)

📋 Key Points

  • XML is strict — every tag closes, attributes are quoted, exactly one root element, tags are case-sensitive. Odoo refuses to load a module with invalid XML.
  • An XML document is a tree: one root, nested children, siblings share a parent. Understanding the tree helps you write XPath expressions correctly.
  • Escape reserved characters: < → &lt;, > → &gt;, & → &amp;. Use CDATA for blocks of code with many special characters.
  • Attributes hold metadata (id, type, name). Element content holds values. Odoo's <record id="..." model="..."> with <field name="...">value</field> children follows this convention throughout.
  • XPath selects nodes in the XML tree. Odoo uses XPath in view inheritance (<xpath expr="..." position="...">) to extend existing views without modifying the original module.
  • Namespaces (xmlns) prevent name collisions. Rare in day-to-day Odoo module dev; you will see them in Odoo's internal RNG schemas and SOAP integrations.
  • Validate XML locally with xmllint --noout file.xml and at runtime with --dev=xml.
  • QWeb adds t-* directive attributes to XML to produce dynamic HTML output for reports, portals, websites, and emails.
  • data/ files load on every install+upgrade; demo/ files load only with demo data. Load order in the manifest is critical.
  • External IDs (XML IDs) let you reference records reliably across modules using ref="module.id" in XML and self.env.ref() in Python.

FAQ

Why does Odoo use XML instead of Python or JSON for views? +

XML was chosen because it maps naturally to a UI tree (nested elements = nested UI components), can be stored in the database as a string (arch field), is diff-friendly for version control, and supports inheritance through XPath — something Python data structures don't do natively.

What is the difference between ref and eval attributes on a <field> element? +

ref="some.xml_id" resolves an external identifier to its database integer ID and assigns that to the field — used for Many2one fields pointing to a known record. eval="expression" evaluates a Python expression at load time — used for lists (ORM commands like [(4, ref('base.group_user'))]), booleans (eval="True"), and computed values.

Can I put Python code inside an Odoo XML file? +

Yes, in two places. The eval attribute runs a Python expression: eval="(datetime.date.today() + relativedelta(days=30)).strftime('%Y-%m-%d')". Server action <field name="code"> fields contain Python code, often wrapped in a CDATA block to avoid escaping <, >, and & throughout the code.

When is noupdate="1" safe vs dangerous? +

Safe for: email templates, sequences, and default configuration records that users are expected to customise — you don't want a module upgrade to overwrite their customisations. Dangerous for: access rights and security rules — if you use noupdate="1" on security records, a later version of your module that fixes a security hole won't be applied on upgrade. Always leave security records in the updatable (default) section.

How do I delete a record using XML? +

Use the <delete> tag with either the XML ID or a domain search:

XML 
<!-- Delete by XML ID -->
<delete id="old_menu_item" model="ir.ui.menu"/>

<!-- Delete by domain (removes matching records) -->
<delete model="ir.ui.menu" search="[('name', '=', 'Legacy Menu')]"/>

This runs on every module upgrade, so only use it to clean up records your module previously created.
