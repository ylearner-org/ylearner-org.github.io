XML Basics for Odoo – Views, Data Files, and QWeb

Odoo uses XML for three critical purposes: defining UI views (form, list, kanban), loading data records (menus, actions, security rules), and writing QWeb templates (PDF reports, portal pages, website pages, email templates). This page covers the XML syntax you need to read and write Odoo XML files confidently, with real examples from the Odoo source.

⏱️ 20 min 🎯 Beginner 📅 Updated 2026

XML Syntax Refresher

XML (Extensible Markup Language) is a strict, tree-structured format. Every Odoo XML file must be valid XML — the parser will reject the file on module load if it is not. Key rules to know before writing Odoo files:

  • Every tag must close: <field name="x"></field> or self-close: <field name="x"/>
  • Attribute values must be in quotes: name="foo" not name=foo
  • XML is case-sensitive: <Field> and <field> are different elements
  • There must be exactly one root element wrapping everything

HTML is lenient — browsers fix missing closing tags and unquoted attributes. XML is strict — <br> is invalid XML; it must be <br/> or <br></br>.

XML 
<?xml version="1.0" encoding="UTF-8"?>
<odoo>
  <!-- This is a comment -->
  <data>
    <record id="some_record" model="some.model">
      <field name="name">Hello Odoo</field>
      <field name="active">True</field>
    </record>
  </data>
</odoo>

A few Odoo-specific conventions to know:

  • Root element is <odoo> — wraps all content in every Odoo XML data file
  • <data noupdate="1"> — records inside this block are created on first install but are not overwritten when the module is upgraded. Use this for base data you expect users to modify (e.g. default email templates, sequences).
  • id attribute on <record> is the XML ID (external identifier) — a unique string used to reference this record from Python, from other XML files, and from view inheritance.

Defining Views in XML

Views are <record> elements with model="ir.ui.view". The arch field contains the actual view structure. Odoo loads these records into the ir.ui.view table when you install or upgrade the module.

Form view example:

XML 
<record id="view_book_form" model="ir.ui.view">
  <field name="name">library.book.form</field>
  <field name="model">library.book</field>
  <field name="arch" type="xml">
    <form string="Book">
      <sheet>
        <group>
          <field name="name" string="Title"/>
          <field name="author_id"/>
          <field name="isbn"/>
          <field name="price"/>
        </group>
        <notebook>
          <page string="Description">
            <field name="description"/>
          </page>
        </notebook>
      </sheet>
    </form>
  </field>
</record>

List (tree) view example:

XML 
<record id="view_book_list" model="ir.ui.view">
  <field name="name">library.book.list</field>
  <field name="model">library.book</field>
  <field name="arch" type="xml">
    <list string="Books">
      <field name="name"/>
      <field name="author_id"/>
      <field name="price"/>
      <field name="active"/>
    </list>
  </field>
</record>

Actions and Menus in XML

A window action tells Odoo which model to open and in which views. Menu items link the navigation bar to these actions.

Window action (opens a model in a view):

XML 
<record id="action_library_book" model="ir.actions.act_window">
  <field name="name">Books</field>
  <field name="res_model">library.book</field>
  <field name="view_mode">list,form</field>
</record>

Menu items:

XML 
<!-- Top-level menu (no parent) -->
<menuitem id="menu_library_root"
          name="Library"
          sequence="10"/>

<!-- Sub-menu linking to action -->
<menuitem id="menu_library_books"
          name="Books"
          parent="menu_library_root"
          action="action_library_book"
          sequence="10"/>

<menuitem> is a shorthand element — unlike views and actions, it does not need a <record> wrapper. Odoo handles the record creation internally.

QWeb Templates

QWeb is Odoo's XML-based template engine. It is used for PDF reports, portal pages, website pages, and email templates. QWeb directives are t-* attributes added to standard HTML/XML elements.

XML 
<template id="report_book_document">
  <t t-call="web.html_container">
    <t t-foreach="docs" t-as="doc">
      <div class="page">
        <h2><t t-esc="doc.name"/></h2>
        <p t-if="doc.author_id">
          Author: <t t-esc="doc.author_id.name"/>
        </p>
        <table>
          <t t-foreach="doc.line_ids" t-as="line">
            <tr>
              <td><t t-esc="line.product_id.name"/></td>
              <td><t t-esc="line.qty"/></td>
            </tr>
          </t>
        </table>
      </div>
    </t>
  </t>
</template>

Key QWeb directives:

DirectivePurpose
t-esc="expr"Output escaped value (safe for user data)
t-raw="expr"Output raw HTML (unsafe — use only for trusted content)
t-if="condition"Conditional rendering
t-else / t-elifElse / else-if branches
t-foreach="list" t-as="item"Loop over iterable
t-call="template.id"Include/extend another template
t-set="var" t-value="expr"Define a variable
t-att-class="expr"Dynamic attribute
t-field="record.field"Render a field with Odoo formatting

Data vs Demo Files

Odoo modules separate production data from demo data:

  • data/ folder — loaded on module install AND every upgrade. Used for menus, actions, views, security, and base configuration records.
  • demo/ folder — loaded ONLY when demo data is enabled at database creation. Used for sample records that help developers and users explore the module.

The __manifest__.py file lists which XML files belong in each category:

Python 
'data': [
    'security/ir.model.access.csv',
    'views/book_views.xml',
    'data/book_data.xml',
],
'demo': [
    'demo/demo_books.xml',
],

Order matters: the security CSV must load before views (views reference model names); views must load before menus; menus must load before actions that reference them. Odoo processes files in the order they are listed in the manifest.

External IDs (XML IDs)

Every <record> element has an id attribute — this is its external identifier. Odoo uses external IDs to reference records across modules without hardcoding database integer IDs (which vary between installations).

  • Within a module, use just the local ID: ref="action_library_book"
  • From another module, use the full qualified form: ref="library_management.action_library_book"
  • In Python: self.env.ref('library_management.action_library_book')

External IDs are also the mechanism for view inheritance — extending existing Odoo views without modifying the original module's code:

XML 
<!-- Add a field to the existing sale.order form view -->
<record id="view_sale_order_form_inherit_library" model="ir.ui.view">
  <field name="name">sale.order.form.library</field>
  <field name="model">sale.order</field>
  <field name="inherit_id" ref="sale.view_order_form"/>
  <field name="arch" type="xml">
    <xpath expr="//field[@name='partner_id']" position="after">
      <field name="library_book_id"/>
    </xpath>
  </field>
</record>

📋 Key Points

  • Odoo XML must be valid XML — strict rules apply; the module will fail to load if the file has XML errors.
  • The root element is always <odoo>. Use <data noupdate="1"> for records that should not be overwritten on upgrade.
  • Views are <record> elements with model="ir.ui.view". The arch field holds the actual view structure.
  • QWeb directives (t-if, t-foreach, t-esc, etc.) power reports, portals, websites, and email templates.
  • The data/ folder loads on install and upgrade; the demo/ folder loads only when demo data is enabled.
  • External IDs (XML IDs) are how Odoo references records reliably across modules and installations — use ref="module.record_id" in XML and self.env.ref() in Python.

FAQ

What is the difference between t-esc and t-raw? +

t-esc HTML-escapes the output — it converts <, >, and & to safe HTML entities. This is safe for user-provided data and prevents XSS attacks. t-raw outputs raw HTML without escaping — use it only for trusted content you fully control. Never use t-raw with user-supplied values.

Can I use HTML in Odoo XML files? +

Odoo XML files must be valid XML. HTML entities like &amp;, &lt;, &gt; are fine. Self-closing tags like <br/> are required — you cannot use bare <br> as you would in HTML. HTML-style loose syntax (unquoted attributes, missing closing tags) will cause XML parse errors and prevent the module from loading.

What does noupdate="1" on a <data> block do? +

Records inside a noupdate="1" block are created on first install but are not overwritten during module upgrades. Use this for base data you expect users to modify — like default email templates or sequence records. Without noupdate="1", every module upgrade would reset those records back to defaults, losing any user customisations.

How do I reference a record from another module in XML? +

Use the full external ID with the module prefix: ref="module_name.record_id". For example, <field name="group_id" ref="base.group_user"/> references the "Internal User" group from the base module. Odoo resolves the XML ID to the database integer ID at load time, so this works reliably across all installations regardless of actual database IDs.
